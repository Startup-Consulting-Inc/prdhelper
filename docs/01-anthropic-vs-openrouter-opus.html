Anthropic API or OpenRouter for Claude Opus: an honest trade-off

Both paths reach the same model. The question is what you give up to get there — and one of them quietly eats your cache hit rate.

By Jaehee Song April 22, 2026 13 min read

THE SHORT VERSION

Use Anthropic's API directly when you need prompt caching that actually sticks, extended thinking, Fast Mode, or zero markup on high-volume workloads. Use OpenRouter when you're prototyping across models, need one key for a small team, want automatic failover, or haven't decided whether Opus is the right model yet. For a production agent that sends the same 40K-token system prompt thousands of times a month — go direct. For everything else, OpenRouter is probably fine.

Last month I ran an experiment. I took a Hermes Agent I'd been building — the kind that loops through tool calls, burns through 40,000-token contexts, and makes hundreds of requests in an afternoon — and I routed it through Claude Opus 4.6 for one day. Just one day. The bill came back at $19.81. That was 65% of my entire 20-day OpenRouter spend, done in a single afternoon.

Was Opus worth it? Sometimes. Was I using it correctly? No, not really. But the bigger question that came out of that experiment is the one I want to answer in this post: when you've decided you want Claude Opus — and you will, eventually, because it's genuinely the best model for certain hard problems — should you call Anthropic's API directly, or route through OpenRouter?

The honest answer depends on what you're optimizing for. Let me show you the trade-offs with real numbers.

01The two paths look identical on paper

Here's the thing people don't realize when they first compare these two options: the token prices are the same. OpenRouter doesn't mark up inference. You pay $5 per million input tokens and $25 per million output tokens either way. That's not a typo — OpenRouter makes its money on a 5.5% credit-purchase fee, not on a per-token cut.

So if cost were the only thing that mattered, this would be a very short blog post. But cost isn't the only thing that matters. Here's a clean side-by-side of what you're actually getting on each path:

What you get Anthropic API (direct) OpenRouter
Base price (Opus 4.6) $5 in / $25 out per MTok $5 in / $25 out per MTok
Platform fee None 5.5% on credit top-ups
Prompt caching Full control, up to 90% savings Works, but passthrough — harder to tune
Batch API (50% off) Yes, native Not available for Anthropic models
Fast Mode (Opus 4.7) Yes, 2.5x faster at 6x price Yes, but auto-routes to Anthropic first-party

Automatic failover No — you handle retries Yes, routes across Bedrock/Vertex/1P

Already you can see the shape of the trade-off. Anthropic gives you more control over the thing you're paying for. OpenRouter gives you more flexibility across what you can reach.

The "one provider" problem nobody talks about

Here's a thing worth pointing out because it surprised me. OpenRouter's big marketing line is provider diversity — route across Bedrock, Vertex AI, Azure, and Anthropic first-party, get the best price automatically. For open-weight models like Kimi K2.5, that genuinely works. There are a dozen providers competing.

For Claude Opus? Not really. Every single one of my 91 Opus calls went through Amazon Bedrock. Not some. Not most. All of them. The "marketplace of providers" I thought I was getting was actually one provider with one price. OpenRouter was adding a network hop, not a negotiation.

THE INDIRECTION TRAP

For premium models like Opus, OpenRouter often has only one provider anyway. You're not getting fallback diversity — you're getting indirection. This isn't OpenRouter's fault; it's a function of how Anthropic licenses Opus to cloud partners. But it does mean the "one key for everything" value proposition weakens considerably the further you move upmarket.

The pricing simulation that convinced me

The bars line up to the penny. For Opus specifically, you will not save money by switching from OpenRouter to Anthropic Direct. The savings live elsewhere — specifically, in how you structure your prompts and whether you're using features like caching and batch at all.

02The cache problem — where the real money hides

If you only read one section of this post, read this one. Prompt caching is where the pricing on Opus actually matters.

Here's the setup: Claude Opus costs $5 per million input tokens. But a cached read costs $0.50 per million tokens — a 90% discount. If you're running an agent that re-sends the same 30,000-token system prompt and tool schema on every call, caching isn't a nice-to-have. It's the difference between a $200 bill and a $20 bill at the end of the month.

What caching does to your Opus bill

Same workload, same prompt, four different cache hit rates

Fresh read ($5/MTok) Cached read ($0.50/MTok) Output ($25/MTok)

Modeled on a typical agent workload: 40K input tokens per call, 500 output tokens, 1,000 calls per day. At 80% cache hits, you save roughly $144/day vs no caching. Over a month, that's more than $4,300.

Here's where the honest trade-off gets uncomfortable. Both paths support prompt caching. But the control you have over it is very different.

On Anthropic's direct API, you add a single cache_control field at the top of your request. You decide what gets cached, for how long (5 minutes or 1 hour), and where the breakpoints sit. You see cache read tokens and cache write tokens as separate line items in your usage. You can tune it.

On OpenRouter, caching happens automatically for supported models. That sounds nice — one less thing to configure. But it also means less visibility. The cache is being managed by the gateway layer on top of the provider, and if your request gets routed to a provider that handles caching differently than Anthropic's first-party endpoint, your hit rate can quietly collapse without anyone telling you.

A REAL PITFALL

On my own OpenRouter data, I hit 38.2% cache utilization on Claude Opus calls. On Kimi K2.5 through SiliconFlow I hit 48.5%. Anthropic's own docs quote cache savings of up to 90%. That 50-point gap isn't the model's fault — it's what happens when you can't see or control where your cache breakpoints are landing.

What a 0% cache hit actually looks like

Let me make this concrete with one evening from my logs. On April 8, between 23:00 and 23:35, my deep-researcher agent made 15 Opus calls in a row. The prompt size ranged from 87K to 100K tokens each. The cache hit rate on every single one of those calls? Zero. I paid full freight for roughly 1.4 million input tokens in 35 minutes — about $7 of pure waste that should have cost me 70 cents with caching on.

Would routing through Anthropic directly have saved me anything? Probably not. The cache doesn't care who's billing you. It cares whether you structured your request with cache_control markers in the right places. I hadn't. The indirection layer wasn't the problem — the prompt architecture was.

WHAT ACTUALLY FIXES THIS

On either path, add cache markers to your stable context blocks. The structure looks the same both ways:

{"type": "text", "text": "Your long system prompt...",
 "cache_control": {"type": "ephemeral"}}

OpenRouter passes this through to Bedrock (or whichever provider is serving you). Anthropic direct consumes it natively. The billing outcome is identical. The savings show up the moment you actually send the marker — which, from looking at my own data, I wasn't doing nearly often enough.

03Latency: the part that matters for users

Pricing isn't the only axis. If you're building a user-facing product, latency eats your UX budget faster than cost eats your margin. And here the picture gets interesting.

From my own 20-day dataset routed through OpenRouter, Claude Opus 4.6 averaged 7.8 seconds total generation time with a 2.5-second time to first token. Those numbers are fine. But OpenRouter specifically routed most of my Opus calls through Amazon Bedrock, which has its own latency profile — not Anthropic's.

If you go direct, you can ask for Anthropic's Fast Mode on Opus 4.6. That's 2.5x faster output at 6x the price. It's absolutely not a default — but if you're building something like a live coding assistant where waiting six seconds for a function signature is a UX disaster, Fast Mode exists specifically for you. OpenRouter supports Fast Mode too, but only by auto-routing to Anthropic first-party when you flag it. Other providers can't serve it.

04The honest pros and cons, side by side

I want to be fair here, because a lot of the trade-off discussion online reads like ad copy for whichever vendor wrote it. Before getting into the specific scenarios where each one wins, here's the unvarnished version.

Anthropic direct API

Why you'd choose it Where it hurts
First-party features — Batch API, Fast Mode, extended thinking, and new model drops arrive on the direct API weeks before they propagate to gateways. Single provider lock-in — if Anthropic has an outage, your system stops. No automatic fallback.
Simpler debugging — one API, one set of error codes, one status page. When something breaks, you know who to ask. Account management overhead — separate billing, separate rate limits, separate keys for dev/staging/prod.
Lower latency floor — one less network hop, no routing layer between you and the model. No model diversity — you get Claude. Need Gemini for vision or Kimi for cheap tool loops? That's a second integration.
Enterprise negotiation — if you're spending real money, Anthropic will talk terms. Tighter regional availability — Anthropic's direct API has fewer regions than Bedrock or Vertex.

OpenRouter

Why you'd choose it Where it hurts
Unified API — one key, one client, one response format for Opus, Sonnet, Gemini, Kimi, and 290+ more. Provider opacity — you don't control which provider handles your request. Price and latency vary silently.
Automatic fallback — if Bedrock is down, OpenRouter can route to Vertex or Anthropic 1P without you writing retry logic. Feature lag — new Anthropic features (cache improvements, computer use updates) often land on OpenRouter days or weeks later.
Cost arbitrage on open-weight models — for Kimi K2.5, switching Chutes → SiliconFlow saves 2.2×. That's a real benefit. No Batch API for Anthropic models — you lose the 50% discount that async workloads would otherwise get.
Unified billing — one invoice, one dashboard, one export. Genuinely nice if you run many models. The "one provider" problem — for premium models like Opus, OpenRouter often has only one provider anyway. You're getting indirection, not diversity.

With that laid out, here's how those trade-offs map to specific situations.

05When OpenRouter is genuinely the better choice

You haven't decided which model to use yet

If you're prototyping, running evaluations, or comparing Opus against Sonnet 4.6, GPT-5, Gemini 3 Pro, and a handful of open-weight models, OpenRouter is the path of least resistance. One key. One billing account. Change a model string, rerun your test. That workflow is genuinely hard to replicate with direct APIs unless you set up four separate accounts.

You want automatic failover

Anthropic occasionally has outages. When Opus on the first-party API is down, OpenRouter can fall back to Opus on Bedrock or Vertex AI automatically. Your agent doesn't stop. If your business depends on Claude being responsive, this alone can justify the 5.5% credit fee.

You're a small team sharing access

OpenRouter's per-key rate limiting, spend caps, and team dashboards are simpler than Anthropic's enterprise setup for teams under about 10 people. You can give your junior developer a key with a $50 monthly cap and not worry about it.

You only pay for what works

This is the quiet win. OpenRouter's "zero-completion insurance" means if a request fails or gets cut off, you don't pay. Anthropic's API bills you for failed attempts the same way it bills for successful ones. At scale, this matters.

06When Anthropic direct is genuinely the better choice

You're sending the same prompt thousands of times

This is where caching control pays for itself in a week. An agent that re-sends a 40K-token system prompt 1,000 times a day is burning roughly $200/day without caching. With properly tuned caching at 80% hit rate, that drops to under $60. You need the visibility that the direct API gives you to hit that 80%.

You need extended thinking on a tight latency budget

Opus 4.6 and 4.7 support a thinking mode where the model reasons through a problem before responding. On the direct API, you can set the thinking budget precisely. Through OpenRouter, it's passed through — but the reasoning_details object needs to be preserved carefully across turns, and the abstraction layer adds a step where things can go wrong.

You process data overnight, not in real-time

Anthropic's Batch API gives you 50% off both input and output. It's async — your results come back within 24 hours. For things like nightly document processing, data enrichment, or eval runs, this turns Opus from "too expensive for that" into "why were we ever paying full price."

THE MATH ON BATCH

Process 10 million input tokens and 2 million output tokens on Opus. Synchronous direct API: $100. Batch API: $50. OpenRouter: $100 (no batch discount on Anthropic models). If your workload can wait a few hours, the direct API cuts the bill in half.

You need compliance, data residency, or audit trails

Enterprise workloads on Opus 4.7 can specify inference_geo: us for US-only routing (with a 1.1x surcharge). That kind of guarantee isn't something a routing layer can give you cleanly — because the routing layer might send your prompt anywhere.

08A practical hybrid pattern

You don't have to pick one. The teams I've seen run this well do something smarter: they use both, for different workloads.

Here's the pattern. Your production agent — the one making 10,000 calls a day with a cacheable prompt — goes straight to Anthropic. Everything else — prototyping, one-off experiments, comparison benchmarks against GPT-5 or Gemini, the occasional Kimi K2 call for a cheap task — routes through OpenRouter. Two API clients, two billing accounts, two very different cost profiles.

# Pseudocode for a hybrid router
def route_call(prompt, model, is_production_agent):
    if model.startswith("anthropic/") and is_production_agent:
        # Go direct — we want cache control and batch access
        return anthropic_client.messages.create(
            model=model.replace("anthropic/", ""),
            system=cached_system_prompt,  # with cache_control
            messages=messages,
        )
    else:
        # Go via OpenRouter — flexibility matters more than control here
        return openrouter_client.chat.completions.create(
            model=model,
            messages=messages,
        )

The added complexity isn't free — you're maintaining two clients, two sets of retry logic, two different observability pipelines. But for workloads at scale, the savings more than cover the engineering time. A team running $500/month on Opus through OpenRouter could probably cut their bill to $150 by moving the production workload direct and tuning their cache.

09What I'd actually tell you to do

Anthropic direct

Pick this when...

  • You're running one repeated agent workload at scale
  • You need cache control to hit 70%+ hit rates
  • You can wait hours for batch jobs (50% off)
  • You need Fast Mode or extended thinking with precise budget control
  • Compliance or data residency is non-negotiable
  • Your monthly Opus spend is already over $500
OpenRouter

Pick this when...

  • You're prototyping or evaluating across multiple models
  • You need one key that also covers GPT-5, Gemini, Kimi, etc.
  • You care about automatic failover more than cache precision
  • Your team needs per-developer keys with spend caps
  • You want zero-completion insurance on flaky workloads
  • Your monthly Opus spend is under $200 and experimentation matters more than optimization
The question isn't "which is better." It's "which is better for this specific workload, this specific team, this specific month." The answer changes as you grow.

10The real lesson from $19.81

Here's what that Apr 8 experiment actually taught me, and it's almost the opposite of what I expected going in. The API gateway is not where Opus gets expensive. The architecture is.

Whether you pay Anthropic directly or OpenRouter indirectly, 91 uncached agent loops with 60K-token prompts will cost roughly the same. The bars in my pricing simulation line up to the penny for a reason — the billing path is a commodity. What varies is what you put through it.

Looking back at that $19.81, almost none of the waste was attributable to OpenRouter. The waste was in:

  • Using Opus for a task that Kimi would have handled equivalently
  • Re-sending 60K tokens of context without any cache markers
  • Letting the agent loop without a spending alert in place
  • Not checking whether my cache hit rate was 40% or 80% until after the bill arrived

Every one of those problems is fixable on either path. Going direct wouldn't have rescued me from any of them. Staying on OpenRouter doesn't doom me to repeat them.

If you're already on OpenRouter and the workflow suits you, keep it. Just don't expect switching APIs to magically make Opus cheap. The price is the price. The waste is in how you use it.

11One last thing about Opus 4.7

Anthropic released Opus 4.7 on April 16, 2026. The sticker price stayed at $5 in / $25 out. But 4.7 uses a new tokenizer that can produce up to 35% more tokens for the same input text — especially on code, structured data, and non-English content. So your Opus bill can genuinely go up 0-35% per request even though Anthropic didn't raise prices.

This matters for both paths. Direct, you'll see it in your usage metrics immediately. Through OpenRouter, you'll see it too, but the abstraction layer means you're one step further from noticing. Either way, if you migrate from 4.6 to 4.7, measure your own token consumption before you commit. And if you're writing Korean or Chinese or any non-English content, the tokenizer change tends to hit harder — something I've seen firsthand on the Korean Brunch posts I publish.

You've got real options here. Both paths lead to the same model. What differs is how much control you want over the road.
